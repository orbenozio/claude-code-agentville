# ממצאי אימות JSONL — Agentville

אימות מעשי של מקור הנתונים, על קבצי JSONL אמיתיים מ-`C:\Users\orben\.claude\projects` (30 קבצים, \~1000 רשומות סוכני-משנה). זה הצעד הראשון שהקונספט דרש — לפני האיפיון.

## TL;DR

* ✅ **המקור עובד.** אפשר לגזור מ-JSONL: כמה סוכנים רצים, מי הם, על מה כל אחד עובד, ומתי התחיל/סיים.
* ✅ **לסוכני-משנה יש קבצי JSONL נפרדים משלהם** — לא צריך לפענח אותם מתוך הסשן הראשי.
* ⚠️ **הכלי נקרא** **`Agent`, לא** **`Task`** (הקונספט הניח `Task`). בגרסת Claude Code הזו השדה הוא `name: "Agent"`.
* ⚠️ **מצב "ממתין לאישור הרשאה" כמעט לא נצפה ב-JSONL** — זו נקודת הסיכון המרכזית לאחד ממצבי-הליבה שהקונספט רצה להציג.

## מבנה הקבצים והתיקיות

```
~/.claude/projects/<path-מקודד>/
├── <session-id>.jsonl              ← הסשן הראשי (הסוכן הראשי)
└── <session-id>/
    └── subagents/
        ├── agent-<agentId>.jsonl   ← קובץ נפרד לכל סוכן-משנה
        └── ...
```

* `<path-מקודד>` = נתיב הפרויקט עם `\`/`:` שהומרו ל-`-` (למשל `c--Users-orben-OneDrive-DEV-Projects-...`).
* כל **סשן** = קובץ ראשי אחד + תת-תיקיית `subagents/` עם קובץ לכל סוכן-משנה שרץ בו.

## סוגי רשומות (שדה `type`) בסשן ראשי

מתוך קובץ ראשי אמיתי: `assistant` (479), `user` (239), ולצידם רשומות תשתית — `queue-operation`, `ai-title`, `mode`, `attachment`, `last-prompt`, `file-history-snapshot`. **לויזואליזציה מספיק להתמקד ב-`assistant`** **ו-`user`**; השאר רעש תשתיתי.

כל רשומה כוללת `timestamp` בפורמט ISO-8601 UTC עם מילי-שניות (`2026-06-03T04:23:58.770Z`) — מצוין ל-tail בזמן אמת ולזיהוי idle לפי "כמה זמן עבר מאז הרשומה האחרונה".

## איך מזהים סוכן-משנה

רשומות בקובץ סוכן-משנה נושאות:

* `isSidechain: true`
* `agentId` — מזהה הסוכן (תואם לשם הקובץ `agent-<agentId>.jsonl`)
* `attributionAgent` — **סוג** הסוכן כמחרוזת קריאה (למשל `"spec-reviewer"`, `"claude-code-guide"`)
* `sessionId` — מקשר לסשן הראשי, וכן `parentUuid`, `cwd`, `gitBranch`

## מיפוי `Agent` → סוכן-משנה (הציר המרכזי של הויזואליזציה)

כשהסוכן הראשי משגר סוכן-משנה, מופיע בסשן הראשי `tool_use` עם:

```jsonc
{ "type": "tool_use", "name": "Agent",
  "input": { "subagent_type": "claude-code-guide",
             "description": "Research /loop, /goal, scheduling",   // ← תווית קצרה מצוינת לבועת דיבור
             "prompt": "..." } }
```

ה-`tool_result` שחוזר (כ-`toolUseResult` ברשומת ה-`user`) סוגר את הלולאה ונותן את **החוליה המקשרת**:

```jsonc
{ "status": "completed",
  "agentId": "a40ce97d...",        // ← מקשר ישירות לקובץ agent-<agentId>.jsonl
  "agentType": "...",
  "totalDurationMs": ..., "totalTokens": ..., "totalToolUseCount": ..., "toolStats": {...} }
```

כלומר: **`description`** **של ה-`Agent`** **→ טקסט בועת הדיבור**, ו-**`agentId`** **שב-`toolUseResult`** **→ קובץ הפעילות החי של אותו סוכן-משנה**. שני אלה יחד הם כל מה שצריך כדי לצייר דמות, לתת לה משימה, ולעקוב אחרי פעילותה.

## נקודות פתוחות לאיפיון (לתשומת לב ה-architect)

1. **מצב "ממתין להרשאה" אינו אמין מ-JSONL בלבד.** אם הוא חובה למוצר — צריך מקור משלים (hook של Claude Code, ניטור תהליך, או ויתור על המצב הזה ב-MVP). שווה החלטה מודעת.
2. **חלון זמן idle** — מהו הסף (שניות?) שמעליו דמות "הולכת הביתה". פרמטר כוונון.
3. **גילוי פרויקטים/סשנים** — האם Agentville מציג את *כל* הפרויקטים בו-זמנית (עיירה אחת גדולה) או נצמד לסשן/פרויקט פעיל אחד. משפיע על מודל ה-tail.
4. **רוטציית קבצים וגודל** — קבצים ראשיים מגיעים ל-MB-ים; tail חייב לקרוא רק את הזנב ולא לפרסר את הכל מחדש בכל פעם.
5. **התאמת גרסאות** — שם הכלי השתנה (`Task`→`Agent`); כדאי שכבת הפרסור תהיה סלחנית לשינויי-סכמה בין גרסאות Claude Code.
